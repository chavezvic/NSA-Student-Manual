# Technical Documentation Guide

## Write for the reader

Identify:

- Who will use the document?
- What task must they complete?
- What do they already know?
- What access and tools do they have?
- What could go wrong?
- What evidence confirms success?

## Recommended procedure structure

1. Title
2. Purpose
3. Audience
4. Scope
5. Prerequisites
6. Safety, security and outage warnings
7. Environment details
8. Procedure
9. Validation
10. Troubleshooting
11. Rollback
12. References
13. Revision history

## Writing rules

- Use direct verbs.
- Put one action in each numbered step.
- State where the action occurs.
- Include expected results after important steps.
- Use exact names for menus, buttons, commands and files.
- Define acronyms on first use.
- Avoid unexplained screenshots.
- Never publish real passwords, tokens or private information.
- Separate commands from output.
- Mark placeholders clearly, such as `<SERVER-NAME>`.

## Screenshot standard

Every screenshot should have:

- A reason for inclusion
- Relevant content only
- Sensitive data removed
- A caption
- A step reference
- Enough context to understand the screen
